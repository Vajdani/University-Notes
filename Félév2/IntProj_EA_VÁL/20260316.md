# api design, rest, endpoints

api - application programming interface 
it defines how requests and responses should be structured and handled 
lots of web applications of this!!!

theyre efficient, enable automation and cooperation between apps, and control security

client -> request -> server 
the request is usually a GET one, it may contain json or other info
some info can also be provided through the url(query param)

api accessibility levels:
1. public
2. partner
3. internal

whoah just like szísárp!

api communication style
1. rest 
uses standard methods(get, post, put, delete) 
stateless 
json or xml
2. soap(simple object access protocal) 
xml 
more secure but "heavier" and slower than rest
3. graphql 
allows clients to request specific data they need 
very dynamic
4. grpc(google remote procedure call) 
uses protobuf, faster than rest
5. xmlrpc
6. jsonrpc

rest principles:
1. client-server
2. stateless
3. cacheable
4. layered system(load balancers, security, etc)
5. uniform interface(standard html requests)
6. resource based urls

safe methods: get, head, options 
idempotent methods(multiple calls = same result): put, delete
non-idempotent methods: post, patch

crud - create read update delete

endpoint - url where an api recieves requests and sends responses 
rest standard is plural endpoint names: user BAD users GOOD

sub resources: 
get /users/x/posts
get /users/x/posts/y
get /posts/y

html put - replaces the entire thing, no partial modification of the resource

html response codes:
- 1xx: info
- 2xx: success
- 3xx: redirect
- 4xx: client error
- 5xx: server error

postman: api testing tool 
it enables you to hand-make or automate requests to test various api endpoints

swagger(swagger.io): designing, building, documenting apis

openapi - restapi standard specification

guzzlehttp: php http client
